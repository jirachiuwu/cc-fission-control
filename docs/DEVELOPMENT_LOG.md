## バグ修正フェーズ（制御以前の土台）

| 症状 | 原因 | 対処 |
|---|---|---|
| 画面が文字化け | CC: Tweaked は日本語フォントを持たない | 画面/ターミナル出力を全て ASCII 化（コメントは日本語のまま）|
| ARM/SCRAM は効くが自動制御・自動更新が死ぬ | フィルタ無し `os.pullEvent` ループが、忙しいサーバー（多 MOD）の**イベントキュー溢れ(256上限)で周期タイマーを取りこぼす** | 制御を `sleep` ベース（フィルタ付き）の専用コルーチンにし、入力処理と `parallel.waitForAny` で並走 |
| 更新が「3分置き」に感じる | peripheral 呼び出しが低 TPS サーバーで重い + モニター同期遅延 | 毎 tick の読み取りを必要最小限に間引き + ヘッダにハートビート（更新間隔の実測表示）で可視化 |
| クラッシュに気づけない | エラーがコンピュータ本体端末にしか出ず、ユーザーはモニターを見ている | 終了/エラーをモニターにも赤字でミラー表示 |

**切り分けの決め手**: `sim/` 以前は「ボタン押すと即反映、自動は死ぬ」というユーザー観察 → 描画は速い＝イベントループの問題、と特定。
さらに `test_monitor.lua`（sleep で数字を数えるだけ）で「コンピュータは tick されている＝環境問題でなくコードの問題」を確定。

## 制御アルゴリズムの進化（v1 → v4.2）

各版の**意図**と**なぜ失敗/前進したか**。

### v1〜v2: 温度ターゲット → 固定 burn 保持
- **意図**: 温度を目標に保つよう burn を比例制御 → メルトダウン。次に「崖を探らず、既知の安全 burn を保持」へ。
- **失敗の本質**: 温度はラグが大きく、冷却の限界（崖）を温度で検知すると手遅れ。固定保持は安全だが保守的すぎ・手動設定が必要。

### v3: auto-seek「coolant≈100% を保つ」
- **意図**: シミュレーター（`sim/`）を導入し、6 戦略をワークフローで競わせて客観評価。隠し上限を変える汎化テストで過学習を排除。
  「coolant を安全ゾーン(≈100%)に保ったまま境界をなぞる」方式を採用。
- **シミュ上は完璧**（全隠し上限で到達率≈100%・波ゼロ）だったが**実機で停止**。
- **失敗の本質（重要な教訓）**: **物理モデルが現実と逆だった。** 実機の coolant は循環で常に 100% 未満（低 burn でも 85% 等）。
  「coolant≥99.5% が健全」という閾値が即発動し、burn 65（温度 417K で全然余裕あり）で止まった。
  → **推測の物理モデルでいくら最適化しても、モデルが間違っていれば実機で破綻する。**

### v4: PI on coolant（低い設定値）
- **転機**: ユーザーが実機物理を説明 — 「coolant は循環で減る、各 burn で安定値に落ち着く、それ以上は減らない」。
- **意図**: モデルを循環物理に作り直し（`sim/bench.py`）。**coolant を低い設定値(12%)に PI 制御**すれば、burn が自動で
  「coolant がその予備分になる点」= 理論上限の直下まで上がって張り付く。slew + アンチワインドアップで起動オーバーシュート抑制。
- **前進**: 実機で初めて burn/温度がちゃんと上がるように。
- **付随バグ**: 統合テストが `checkSafety` の coolant SCRAM 閾値 25% が運転点 12% より高く**正常運転中に SCRAM 発振**するのを発見 → 3% に下げ + 積分器の外部同期を追加。

### v4.1: 変化率ゲート（HOLD while falling）
- **症状**: 登るが「coolant が落ちてる最中も追加・追加・追加」して行き過ぎる（復水ラグでの overshoot）。
- **意図**: **coolant が落下中（過渡）は積分を止めて HOLD、落ち着いたら上げる**。= 人間の「状態を見ながら上げる」の自動化。
- **検証**: ベンチで復水を遅く（kSettle 小）するほど旧 PI が行き過ぎることを再現 → ゲートで行き過ぎが激減。

### v4.2: 二重制約（coolant 下限 + heated 上限）— 現行
- **転機**: ユーザー観察 —「蒸気がタービンにパンパン → ボイラーから水が消える」。**本当の律速はタービン**だと判明。
- **意図**: モデルにタービン上限 `Bt` を追加（heated が Bt で飽和 → ボイラー水枯れ → coolant 道連れ）。
  制御を **`err = min(coolant 余裕, heated 余裕)`** にして、coolant 低下 OR heated 上昇の**先に効く方**で頭打ち。
- **検証**: 実 reactor.lua の統合テストで、冷却材律速 / タービン律速 / 均衡 の全構成で「先に詰まる方」で HOLD・溶融なし・発振なしを確認。
  タービン律速では heated を 60% に保ち、蒸気がパンパンになる前に止まる。

## 教訓（このプロジェクトで効いた原則）

1. **物理を推測で決めない。** v3 の失敗は「もっともらしい物理モデル」を信じたこと。実機の挙動はユーザー実測 or 現物確認で固める。
2. **ゲーム外シミュレーターが効く。** ロジックの暴走・発振・収束不能・行き過ぎは `sim/` で事前に露見する。実機の試行錯誤コスト（メルトダウン）を払わずに詰められる。ただし**モデルが正しい範囲でしか正しくない**（v3 の戒め）。
3. **汎化テスト。** 隠しパラメータ（runout 上限・タービン上限・復水ラグ）を変えた複数条件で検証し、特定構成への過学習を排除。
4. **症状の言語化がそのまま設計の手がかり。** 「ボタンは即反映」「追加追加追加」「蒸気がパンパン」等、ユーザーの観察が毎回ブレークスルーになった。
5. **デプロイ確認を仕組みに。** `version` を画面表示。「直したのに変わらない」の多くは『ゲーム内で install.lua を流し直してない』だった。

## いまだ粗い点 / 次にやれること（refinement backlog）

現状で「とりあえず動く」が、ユーザー評: **「まだ詰められるくらい、粗い」**。改善候補:

- **タービン蒸気を直接読む**: 今は reactor の `heatedPct`（逆流蒸気）でタービン詰まりを間接検知。Industrial Turbine の
  `getSteamFilledPercentage`（あれば）を `turbine.lua` で読み、3 つ目の制約に加えるとより早く・正確に頭打ちできる。
- **setpoint の自動チューニング**: `coolantSetpoint`/`heatedSetpoint` を手動既定（12/60）でなく、運転中に
  「安全に維持できる最小 coolant / 最大 heated」を学習して詰める（より理論上限に近づける）。
- **ゲインの最適化**: `piKi/Kp`、`slew`、`settleTol` は手調整。ベンチでの探索（ワークフローの設計パネル方式）でさらに波を詰められる。
- **ハンチングの低減**: 二重制約の切り替わり点付近での微小な揺れをさらに滑らかに（デッドバンド/ヒステリシス）。
- **物理モデルの精緻化**: 現 `sim/bench.py` のモデルは一次遅れ + 線形平衡の近似。実測カーブ（coolant%(burn), heated%(burn)）を
  ユーザーから採って当てはめると、シミュの予測精度＝チューニングの当たりが上がる。
- **複数炉 / SCADA 化**: 参考にした cc-mek-scada のようにマルチ炉・GUI 強化。
